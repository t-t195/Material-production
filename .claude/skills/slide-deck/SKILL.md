# コンサル級提案書ジェネレーター

スライドHTMLの作成 → ビルド → 検証・修正までをこのセッション内で完結させる。
**これは判断の余地を残さないための手順書である。省略・アレンジせず、上から順に実行すること。**

## 絶対ルール（作業前に必ず確認）

1. **ゼロから書かない** — 必ず `components/` の見本HTMLをコピーして文言を差し替える
2. **templates/ は編集禁止** — `base.css`・`components.css` の変更は全案件に影響する。
   案件固有の調整はスライドHTML内の `style` 属性で行う
3. **色・フォントを増やさない** — 使ってよい色は `base.css` の CSS 変数のみ
4. **外部リソース禁止** — 画像・CDN・Webフォント・チャートライブラリを使わない。
   図形は CSS のみで描画する（`clip-path` / `border` / `div` の高さ・位置指定）
5. **検証を省略しない** — Step 5 の PNG 目視は必須。ビルド成功＝完成ではない

## Step 0. 環境確認（セッション初回のみ）

**以降のコマンド・相対パスはすべてリポジトリルート `C:\dev\資料作成全般` を基準にする。**
カレントディレクトリが異なる場合は最初に移動すること。

```powershell
Set-Location C:\dev\資料作成全般
Test-Path node_modules                                  # False なら npm install
Test-Path "$env:LOCALAPPDATA\ms-playwright\chromium-*"  # False なら npx playwright install chromium
```

両方の Test-Path が True になったことを確認してから Step 1 へ進む。
インストールがエラーで終了した場合は、エラー内容をユーザーに報告して指示を仰ぐ。

## Step 1. 要件の確定

以下が揃っていなければ、推測で進めずユーザーに確認する:
- 案件名（`slides/` のフォルダ名になる。例: `2026-07-abc-teian`）
- 宛先（「株式会社◯◯ 御中」）と機密表記（既定は「貴社限り」）
- 各スライドの: タイトル ／ メッセージ行（そのスライドの主張1文）／
  本文の要素（表の行・箇条書き・数値）／ 出所

参考スクリーンショットが渡されたら Read で開き、「どの部品に対応するか」を先に分解する。

## Step 2. 部品の選定（決定表）

| 伝えたい内容 | コピー元 |
|---|---|
| 表紙 | components/00-cover.html |
| 業務・課題の構造整理 | components/01-structure-table.html |
| 工程・スケジュールと実施範囲 | components/02-process-chevron.html |
| 課題→ありたい姿→効果の論理 | components/03-logic-frame.html |
| 投資対効果・回収シミュレーション | components/04-roi-bridge.html |
| 提供価値の一覧 | components/05-value-definition.html |
| フェーズ・項目別の詳細表 | components/06-phase-detail.html |
| 章の区切り・目次 | components/07-section-divider.html |
| 複数案の比較・推奨 | components/08-comparison.html |
| 想定問答 | components/09-faq.html |

該当する部品が無い場合のみ新規作成する。その場合も最も近い部品をコピーして改造する
（資産化の手順は Step 5.5 に従う）。

## Step 3. スライドHTMLの作成

1. `slides/<案件名>/` を作成する
2. 部品HTMLを `NN-<内容>.html` としてコピーする（`00-cover.html` から連番。
   ファイル名の昇順＝スライド順）
3. **CSSパスを書き換える（コピー直後に必ず実施）**:
   `href="../templates/` → `href="../../templates/`
   （components/ と slides/<案件名>/ では階層が1つ違う。
   忘れるとスタイルが一切当たらず「白紙のような素のHTML」になる）
4. 文言を差し替える: タイトル／メッセージ行／本文／出所行／ページ番号／宛先・機密表記
   - **採番規則**: 表紙にはページ番号を付けない（ただし表紙を1ページ目と数える）。
     2枚目以降の `.page-no` を 2 から連番で振る。章扉にもページ番号は付ける
   - 表紙・章扉に見出し・メッセージ行・出所行が無いのは仕様（付け足さない）
5. 文字量の目安（超える場合はフォントを小さくするのではなく、内容を要約するか2枚に分割する）:
   - メッセージ行: 全角60字以内の1文
   - 表のセル: 2行以内 ／ 箇条書き: 1項目40字以内・1ブロック5項目以内
6. 表系部品（01・05・06・08）は高さ100%で行が均等に伸びる設計。行数が少なく間延びする
   場合のみ `<table class="..." style="height:auto">` で上書きしてよい（高さ調整はこの方法だけ）

## Step 4. ビルド

```powershell
npm run build -- slides/<案件名>
```

- `⚠ はみ出し検知` → 該当スライドの文言を短縮するかスライドを分割して再ビルド。
  **フォントサイズを縮小して収める対応は禁止**（投影時の可読性が下がる）
- `⚠ 共通フレーム未適用` → `.slide` の外枠が無い。部品のコピーからやり直す
- 警告ゼロになるまで繰り返す。3回の修正で解決しない場合は、状況を整理して
  ユーザーに報告し指示を仰ぐ

## Step 5. 目視検証（省略禁止）

`output/<案件名>/png/` の全PNGを Read で開き、各スライドが以下の**合格条件**を
すべて満たすか確認する:

- [ ] 文字の見切れ・要素の重なりが無い
- [ ] 見出し・メッセージ行・出所行・ページ番号・スタンプの位置が全スライドで揃っている
      （表紙・章扉に見出し・メッセージ行・出所行が無いのは仕様なので除外）
- [ ] ページ番号が採番規則（Step 3-4）どおり連番になっている
- [ ] 誤字・不自然な位置での改行が無い
- [ ] スライド下半分だけ空白のような偏りが無い
- [ ] メッセージ行の主張と本文の中身が対応している

**判定と遷移:**
- 満たさない項目が1つでもあるスライド → 該当する `slides/<案件名>/` の HTML を
  Step 3 のルール（3-3〜3-6）に従って修正し、Step 4 の再ビルドへ戻る。
  部品ごと差し替える場合は Step 3-2 から再実施する
- 全スライドが全項目を満たした → Step 5.5 へ進む

## Step 5.5. 資産化（新規部品を作った場合のみ。無ければ Step 6 へ）

1. 新規部品の文言をサンプル（架空の一般的な内容）へ戻した見本HTMLを
   `components/NN-<部品名>.html`（既存の続き番号）として保存する
2. `components/index.html` の既存の `<div class="card">…</div>` 行を1行コピーし、
   ファイル名とラベルを差し替えて追記する
3. `npm run catalog` を実行し、警告ゼロで完了することを確認する

## Step 6. 完了報告

以下をすべて含めて報告する:
- 出力パス: `output/<案件名>/<案件名>.pptx`（投影用）と `.pdf`（事前送付用）
- 検証結果（Step 5 のチェック結果と、途中で修正した内容）
- 未解決の懸念（無ければ「なし」と明記）
- 未コミットの変更が残っていれば commit と push を提案する

## トラブルシューティング（決定表）

| 症状 | 原因と対処 |
|---|---|
| スライドが白紙・素のHTMLに見える | CSSパスの階層ミス → Step 3-3 を再実施 |
| Playwright 起動エラー | Chromium 未導入 → `npx playwright install chromium` |
| 表が縦に間延びする | 行数不足 → 行を増やす、または `style="height:auto"`（Step 3-6） |
| 文字が枠からあふれる | 文言を短縮 or スライド分割。フォント縮小はしない |
| 新しい図形・グラフが必要 | CSS のみで描画。画像ファイル・外部ライブラリは使わない |
| PPTX の文言を直したい | PPTX は手編集しない（正本はコード）→ HTML を直して再ビルド |
